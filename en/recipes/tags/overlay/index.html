Overlay для RS485 на UART1 с GPIO управлением направления

November 29, 2025 · 2 min read
dmn
maintainer

Создание overlay для настройки RS485 на UART1 (NAPI-C) с контролем направления передачи через GPIO2_B4.

Overlay файл uart1-de.dts​

/dts-v1/;
/plugin/;

/ {
  compatible = "rockchip,rk3308";

  fragment@0 {
    target = <&uart1>;
    __overlay__ {
      status = "okay";

      pinctrl-names = "default";
      pinctrl-0 = <&uart1_xfer>;

      linux,rs485-enabled-at-boot-time;
      rs485-rts-active-high;
      rs485-rts-delay = <0 0>;
      rts-gpios = <&gpio2 12 0>;
    };
  };
};

Разбор параметров​

Основные настройки:​

  • status = "okay" - включает UART1
  • pinctrl-0 = <&uart1_xfer> - настройки пинов для TX/RX

RS485 специфичные параметры:​

  • linux,rs485-enabled-at-boot-time - включает RS485 при загрузке
  • rs485-rts-active-high - RTS активен высоким уровнем
  • rs485-rts-delay = <0 0> - задержки переключения [before, after] в микросекундах
  • rts-gpios = <&gpio2 12 0> - GPIO для управления направлением (DE pin)

Расчет GPIO​

Для GPIO2_B4:

  • Банк: 2
  • Группа: B (8-15)
  • Пин: 4
  • Итого: 8 + 4 = 12

Установка overlay​

Компиляция и установка:​

# Компиляция
dtc -@ -I dts -O dtb -o uart1-de.dtbo uart1-de.dts

# Копирование в системную папку
sudo cp uart1-de.dtbo /boot/dtb/overlay/

# Добавление в armbianEnv.txt
echo "user_overlays=uart1-de" >> /boot/armbianEnv.txt

# Перезагрузка
reboot

Автоматический способ (Armbian):​

armbian-add-overlay uart1-de.dts

Проверка работы​

Проверка устройства:​

ls -la /dev/ttyS1

Проверка RS485 параметров:​

# Через device tree
ls -la /proc/device-tree/serial@ff0b0000/
cat /proc/device-tree/serial@ff0b0000/status

# Должны присутствовать файлы:
# linux,rs485-enabled-at-boot-time
# rs485-rts-active-high
# rs485-rts-delay
# rts-gpios

Проверка GPIO:​

gpioinfo -c gpiochip2 | grep "line  12"

Настройка порта​

Установка параметров:​

stty -F /dev/ttyS1 9600 cs8 -cstopb -parenb -ixon -ixoff -crtscts

Проверка настроек:​

stty -F /dev/ttyS1 -a

Тестирование RS485​

Отправка данных:​

echo "test message" > /dev/ttyS1

Мониторинг GPIO:​

# Наблюдение за переключением DE pin
watch -n 0.1 "gpioget -c gpiochip2 12"

Логирование трафика:​

# Отправка с мониторингом
stdbuf -oL cat /dev/ttyS1 | tee rs485.log &
echo "test" > /dev/ttyS1

Альтернативные GPIO​

Если GPIO2_B4 занят, можно использовать:

rts-gpios = <&gpio2 11 0>;  // GPIO2_B3
rts-gpios = <&gpio2 13 0>;  // GPIO2_B5
rts-gpios = <&gpio3 27 0>;  // GPIO3_B3

Отладка проблем​

Проверка конфликтов пинов:​

cat /sys/kernel/debug/pinctrl/pinctrl-rockchip-pinctrl/pinmux-pins | grep gpio2

Логи ядра:​

dmesg | grep uart
dmesg | grep rs485

Данный overlay обеспечивает корректную работу RS485 интерфейса с автоматическим управлением направлением передачи через выделенный GPIO пин.

Получение текущего DTS из загруженной системы

November 23, 2025 · 2 min read
dmn
maintainer

Как получить Device Tree Source (DTS) из работающей системы для анализа конфигурации оборудования.

Основная команда​

dtc -I fs -O dts /sys/firmware/devicetree/base > /tmp/running.dts

Эта команда создает файл /tmp/running.dts с полным описанием дерева устройств текущей системы.

Что такое running.dts​

Running DTS - это ключевой файл, показывающий как система "понимает" всю периферию и интерфейсы:

  • ✅ Активированные устройства и интерфейсы
  • ✅ Примененные оверлеи
  • ✅ Конфигурацию пинов и GPIO
  • ✅ Параметры тактирования
  • ✅ Адреса устройств в памяти

Анализ файла​

Просмотр структуры:​

# Открыть в текстовом редакторе
nano /tmp/running.dts

# Поиск конкретного устройства
grep -A 10 -B 5 "uart" /tmp/running.dts

# Поиск GPIO конфигурации  
grep -A 5 -B 5 "gpio" /tmp/running.dts

Полезные паттерны поиска:​

# UART интерфейсы
grep -n "serial@" /tmp/running.dts

# I2C контроллеры
grep -n "i2c@" /tmp/running.dts  

# SPI интерфейсы
grep -n "spi@" /tmp/running.dts

# GPIO банки
grep -n "pinctrl@" /tmp/running.dts

Практическое применение​

1. Отладка проблем с устройствами​

# Проверить статус UART1
grep -A 20 "serial@ff0b0000" /tmp/running.dts | grep status

2. Анализ пин-конфигурации​

# Посмотреть назначение пинов
grep -A 10 "pinmux" /tmp/running.dts

3. Создание оверлеев​

Используйте running.dts как основу для создания пользовательских оверлеев.

Интеграция с AI​

🤖 Совет: Этот DTS можно "скормить" ChatGPT и:

  • Задавать вопросы о конфигурации
  • Просить создать overlay под ваши нужды
  • Анализировать конфликты устройств

Пример запроса к ChatGPT:​

Проанализируй этот DTS файл и покажи, какие UART интерфейсы активны:
[вставить содержимое running.dts]

Сравнение конфигураций​

До и после изменений:​

# Сохранить "до"
dtc -I fs -O dts /sys/firmware/devicetree/base > /tmp/before.dts

# Применить изменения (overlay, reboot)

# Сохранить "после" 
dtc -I fs -O dts /sys/firmware/devicetree/base > /tmp/after.dts

# Сравнить
diff /tmp/before.dts /tmp/after.dts

Формат вывода​

Структура DTS:​

/ {
    compatible = "rockchip,rk3308";
    
    serial@ff0b0000 {
        compatible = "rockchip,rk3308-uart", "snps,dw-apb-uart";
        reg = <0x0 0xff0b0000 0x0 0x100>;
        status = "okay";
    };
}

Альтернативные команды​

Просмотр отдельных узлов:​

# Конкретное устройство
cat /sys/firmware/devicetree/base/serial@ff0b0000/status

# Совместимость
cat /sys/firmware/devicetree/base/compatible

Бинарный формат:​

# Получить DTB
cp /sys/firmware/fdt /tmp/running.dtb

Этот метод - незаменимый инструмент для понимания того, как система интерпретирует аппаратную конфигурацию, и основа для создания правильных оверлеев.

Компиляция ядра NapiLinux - быстрый метод

October 22, 2025 · 2 min read
dmn
maintainer

🔥 Быстрый метод компиляции только ядра NapiLinux без пересборки всей системы.

Предупреждение​

☝️ Это "грязный" быстрый метод для тестирования изменений ядра.
💡 Правильный способ - компиляция через Yocto.

1. Клонирование репозитория​

Выберите нужную ветку для вашей платформы:

Доступные ветки:

  • ✔️ rk-6.1 - для RK3308
  • ✔️ orange-pi-6.6 - для Orange Pi
  • ✔️ rk35xx - для RK35xx
  • И другие (см. GitLab)
git clone https://gitlab.nnz-ipc.net/pub/napilinux/kernel.git --branch=rk-6.1 --depth=1
cd kernel/

2. Настройка окружения для ARM64​

export ARCH=arm64
export CROSS_COMPILE=aarch64-linux-gnu-

3. Подготовка конфигурации​

Конфиги находятся в папке /arch/arm64/configs. Выберите нужный:

Для устройств NAPI:

make napi_defconfig

Для CM4:

make opi_cm4_defconfig

После выполнения появится файл .config, который можно отредактировать под свои нужды.

4. Компиляция ядра​

make -j10

Где 10 - количество параллельных процессов (подберите под свою систему).

5. Результаты сборки​

После успешной компиляции получите:

  • ✔️ Образ ядра: arch/arm64/boot/Image
  • ✔️ Главная DTB: arch/arm64/boot/<файл>.dtb
  • ✔️ Оверлеи: arch/arm64/boot/dts/rockchip/

6. Установка в систему​

Важно про overlay root!​

NapiLinux использует root (/) как виртуальную партицию overlay. Просто скопировать файлы не получится - u-boot их не увидит!

Решения:​

Работа с SD-картой:

# Извлеките SD-карту
# Подключите к компьютеру
# Примонтируйте физические партиции
sudo mount /dev/sdX1 /mnt/boot
sudo cp arch/arm64/boot/Image /mnt/boot/
sudo cp arch/arm64/boot/dts/rockchip/*.dtb /mnt/boot/
sudo umount /mnt/boot

Работа с NAND/eMMC:

# Загрузитесь с SD-карты  
# Примонтируйте целевые партиции вручную
lsblk -f  # определите партиции
sudo mount /dev/mmcblk0p1 /mnt/boot
sudo cp arch/arm64/boot/Image /mnt/boot/
sudo cp arch/arm64/boot/dts/rockchip/*.dtb /mnt/boot/
sudo umount /mnt/boot

7. Диагностика партиций​

Посмотреть структуру партиций:

lsblk
lsblk -f

Дополнительные настройки​

Кастомизация конфига:​

# Интерактивное меню
make menuconfig

# Текстовое меню  
make nconfig

Сборка только DTB:​

make dtbs

Сборка модулей:​

make modules
make modules_install INSTALL_MOD_PATH=/path/to/rootfs

Отладка проблем​

Проверка версии компилятора:​

${CROSS_COMPILE}gcc --version

Очистка сборки:​

make clean
make mrproper

Проверка конфига:​

make ARCH=arm64 listnewconfig

Данный метод позволяет быстро тестировать изменения ядра без полной пересборки дистрибутива через Yocto.

Перенос blue-led на GPIO2B4 в Armbian

September 17, 2025 · 2 min read
dmn
maintainer

Исправление предыдущего примера переноса индикатора: GPIO3B3 конфликтует с SPI1-CLK, поэтому переносим blue-led на GPIO2B4.

Проблема с предыдущим решением​

В предыдущем примере GPIO3B3 попал на SPI1-CLK, что может вызвать конфликты с SPI интерфейсом. Используем более безопасный GPIO2B4.

Создание правильного overlay​

Файл blue-led-gpio2b4-1.dts:​

/dts-v1/;
/plugin/;

/ {
    fragment@0 {
        /* Целевой узел именно под /leds */
        target-path = "/leds/blue-led";
        __overlay__ {
            gpios = <&gpio2 12 0>;          /* GPIO2_B4, ACTIVE_HIGH */
            linux,default-trigger = "heartbeat";
            default-state = "on";
            status = "okay";
        };
    };
};

Расчет номера GPIO​

Для GPIO2B4:

  • Банк: 2
  • Группа: B (8-15)
  • Пин: 4
  • Итого: 8 + 4 = 12

Установка overlay​

Автоматический способ:​

armbian-add-overlay blue-led-gpio2b4-1.dts

Ручной способ:​

# Компиляция
dtc -@ -I dts -O dtb -o blue-led-gpio2b4-1.dtbo blue-led-gpio2b4-1.dts

# Копирование в системную папку
sudo cp blue-led-gpio2b4-1.dtbo /boot/dtb/overlay/

# Добавление в конфигурацию
echo "user_overlays=blue-led-gpio2b4-1" >> /boot/armbianEnv.txt

Применение изменений​

reboot

Проверка результата​

Визуальная проверка:​

  • Диод должен начать мигать в режиме heartbeat на пине GPIO2B4

Программная проверка:​

# Проверка активных LED
ls -la /sys/class/leds/

# Проверка триггера
cat /sys/class/leds/blue-led/trigger

# Проверка GPIO
gpioinfo -c gpiochip2 | grep -A5 -B5 "line  12"

Преимущества GPIO2B4​

✅ Безопасность:

  • Не конфликтует с SPI интерфейсами
  • Не используется системными функциями
  • Доступен на разъеме

✅ Совместимость:

  • Работает со всеми версиями Armbian
  • Не требует дополнительных драйверов

Отладка проблем​

Проверка загрузки overlay:​

dmesg | grep overlay | grep blue-led

Проверка конфликтов GPIO:​

cat /sys/kernel/debug/gpio

Возврат к умолчанию:​

# Удаление из конфигурации
sed -i '/blue-led-gpio2b4-1/d' /boot/armbianEnv.txt
reboot

Альтернативные пины​

Если GPIO2B4 занят, можно использовать:

  • GPIO2B3 (11) - <&gpio2 11 0>
  • GPIO2B5 (13) - <&gpio2 13 0>
  • GPIO2C0 (16) - <&gpio2 16 0>

Выберите свободный пин в соответствии с вашей схемой подключения.

Данное решение обеспечивает корректную работу heartbeat индикатора без конфликтов с системными интерфейсами.

Перенос heartbeat диода на пользовательский GPIO через overlay

September 15, 2025 · 2 min read
dmn
maintainer

Руководство по переносу индикатора активности (heartbeat LED) с встроенного GPIO на свободный пин с помощью пользовательского overlay в Armbian.

Задача​

По умолчанию heartbeat диод находится на GPIO0A5, которого нет на "гребёнке" NAPI. Необходимо перенести его на GPIO3B3, который доступен на разъеме и не занят.

Создание пользовательского overlay​

1. Создание файла DTS​

Создаем файл blue-led-gpio3b3-2.dts:

/dts-v1/;
/plugin/;

/ {
    fragment@0 {
        /* Целевой узел для LED */
        target-path = "/leds/blue-led";
        __overlay__ {
            gpios = <&gpio3 27 0>;          /* GPIO3_B3, ACTIVE_HIGH */
            linux,default-trigger = "heartbeat";
            default-state = "on";
            status = "okay";
        };
    };
};

2. Добавление overlay в систему​

Используем встроенную команду Armbian:

armbian-add-overlay blue-led-gpio3b3-2.dts

Что делает эта команда:

  • Автоматически компилирует .dts в .dtbo
  • Перемещает файл в нужную папку
  • Добавляет запись в armbianEnv.txt

3. Применение изменений​

reboot

Проверка результата​

После перезагрузки:

  1. Визуальная проверка - диод должен начать мигать на новом пине
  2. Проверка активного GPIO:
# Найти активный LED
find /sys/class/leds -name "*blue*" -o -name "*heartbeat*"

# Посмотреть какой GPIO используется
cat /sys/class/leds/*/device/modalias

Расчет номера GPIO​

Для GPIO3B3:

  • Банк: 3
  • Группа: B (8-15)
  • Пин: 3
  • Итого: 8 + 3 = 11 (но в DTS указываем 27 = 3*8 + 3)

Альтернативный способ - ручное редактирование​

Проверка текущей конфигурации:​

cat /boot/armbianEnv.txt | grep overlay

Ручное добавление:​

echo "user_overlays=blue-led-gpio3b3-2" >> /boot/armbianEnv.txt

Отладка проблем​

Проверка загрузки overlay:​

dmesg | grep overlay
dmesg | grep gpio

Проверка доступных LED:​

ls -la /sys/class/leds/

Проверка триггеров:​

cat /sys/class/leds/*/trigger

Важные замечания​

  • GPIO3B3 может пересекаться с SPI1-CLK - проверьте конфликты
  • Overlay применяется только после перезагрузки
  • Для отмены изменений удалите строку из armbianEnv.txt и перезагрузитесь

Этот метод позволяет легко переназначать системные индикаторы на удобные пины без пересборки ядра.

Конфигурация uEnv.txt для Сборщик компакт 0.3

August 13, 2025 · One min read
dmn
maintainer

Конфигурация загрузчика для модуля "Сборщик компакт 0.3" с блоком питания 1.2А.

Содержимое uEnv.txt​

Файл /boot/uEnv.txt для правильной работы устройства:

verbosity=7
fdtfile=rk3308-napi-c.dtb
console=ttyS0,115200n8
overlays=rk3308-uart1 rk3308-uart3-m0 rk3308-i2c1-ds1338 rk3308-i2c3-m0 rk3308-usb20-host
kernelimg=Image
extraargs=

Описание параметров​

Основные настройки:​

  • verbosity=7 - максимальный уровень подробности загрузки
  • fdtfile=rk3308-napi-c.dtb - файл описания устройства для RK3308
  • console=ttyS0,115200n8 - настройки консольного интерфейса
  • kernelimg=Image - имя файла ядра

Активированные оверлеи:​

  • rk3308-uart1 - UART1 интерфейс
  • rk3308-uart3-m0 - UART3 в режиме m0
  • rk3308-i2c1-ds1338 - I2C1 для часов реального времени DS1338
  • rk3308-i2c3-m0 - I2C3 в режиме m0
  • rk3308-usb20-host - USB 2.0 хост контроллер

Важная информация о RS485​

🔥 Важно: У данной модели интерфейс RS485 находится на /dev/ttyS3

Проверка RS485:​

# Проверка наличия устройства
ls -la /dev/ttyS3

# Проверка параметров порта
stty -F /dev/ttyS3

Применение конфигурации​

  1. Скопируйте конфигурацию в файл /boot/uEnv.txt
  2. Перезагрузите систему
  3. Проверьте работу интерфейсов

В Токосборщике на модуле CM4 UART порты имеют специальное назначение и требуют правильной настройки оверлеев.

Назначение UART портов​

✔️ UART3 - внешний датчик Modbus
✔️ UART9 - модуль расширений (Zigbee)
✔️ UART7 - встроенный датчик тока

Конфигурация оверлеев​

Для корректной работы в Debian необходимо подключить оверлеи с uart7,9 в файле /boot/orangepiEnv.txt:

verbosity=1
bootlogo=false
extraargs=cma=128M
overlay_prefix=rk356x
overlays=uart7-m2 uart9-m2
rootdev=UUID=a0f8ca89-7eb7-4a1e-947a-2341637b4782
rootfstype=ext4
console=serial

Проверка конфигурации​

Просмотр текущей конфигурации:

cat /boot/orangepiEnv.txt

Проверка активации​

После перезагрузки проверьте наличие устройств:

ls -la /dev/ttyS*

Должны появиться соответствующие устройства:

  • /dev/ttyS3 - для внешнего Modbus датчика
  • /dev/ttyS7 - для встроенного датчика тока
  • /dev/ttyS9 - для модуля расширений Zigbee

Эта конфигурация обеспечивает корректную работу всех интерфейсов в системе Токосборщика на базе CM4.

Подключение UART4 в NapiLinux

dmn
maintainer

Для подключения UART4 в NapiLinux необходимо выполнить несколько простых шагов.

Пошаговая инструкция​

1. Создание папки для пользовательских оверлеев​

mkdir -p /boot/overlay-user

2. Размещение файла оверлея​

Положите в созданную папку файл rk3308-uart4.dtbo:

# Убедитесь, что файл находится в нужном месте
ls -la /boot/overlay-user/rk3308-uart4.dtbo

3. Настройка загрузки​

Добавьте строчку в файл /boot/uEnv.txt:

echo "user_overlays=rk3308-uart4" >> /boot/uEnv.txt

4. Перезагрузка системы​

reboot

5. Проверка результата​

После перезагрузки должно появиться устройство /dev/ttyS4:

ls -la /dev/ttyS4

Проверка работоспособности​

Для проверки работы UART4 можно использовать:

# Проверка доступности устройства
stat /dev/ttyS4

# Отправка тестовых данных
echo "test" > /dev/ttyS4

Данный метод позволяет быстро активировать дополнительный UART интерфейс в системе NapiLinux без необходимости пересборки ядра.
